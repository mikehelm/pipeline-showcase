Two Approaches, One Goal

‚ö° Specification-Driven (Otto √ó Mike)

  • Invests upfront in a near-perfect spec
  • One shot execution ‚Äî spec so good Codex rarely needs a second pass
  • Multi-model orchestration (4 AI models, each in their role)
  • Review loop catches flaws BEFORE code exists
  • Produces 30-70KB PRDs as living documentation
  • Screenshot-driven visual QA closes the loop
  • Best for: known problem ‚Üí optimal solution

üîÄ /spawn-team (Serhii)

  • Explores N approaches in parallel
  • "Must NOT" constraints force genuine diversity
  • Git worktrees = real isolation between attempts
  • Automated evaluation (lint, tests, diff size) picks winner
  • Produces COMPARISON.md documenting why winner won
  • Rollback available at any point
  • Best for: ambiguous problem ‚Üí discover the right approach

Use our spec-first approach for the "what" ‚Äî deep research, perfect PRD, review to 9/10.
Use /spawn-team for the "how" ‚Äî when multiple valid implementation approaches exist, let them race.

The spec eliminates decision ambiguity at the architecture level. Spawn-team eliminates implementation ambiguity at the code level. Together: zero ambiguity at every layer.

Example: "Build an auth system" ‚Üí Our PRD defines exact token flow, session schema, and error states. Spawn-team tries three implementations: hand-rolled JWT vs NextAuth vs server sessions. Winner gets merged, losers document why they lost.
